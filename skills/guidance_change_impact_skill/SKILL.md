# Guidance Change Impact Skill

## 数据依赖与执行前置条件

本技能依赖 Wind 数据源。执行本技能前，必须先检测 `wind-mcp-skill` 是否已安装且可用。

可用性判断至少包括：

- `wind-mcp-skill` 已安装；
- 其 CLI / MCP 调用入口可执行；
- `WIND_API_KEY` 已配置；
- 能完成一次最小 Wind 查询并返回有效数据。

若 `wind-mcp-skill` 已安装且可用，先通过该技能获取并整理行情、K 线、成交量、换手率、指数环境、板块表现、公告新闻等必要数据，再进入分析或筛选。

若未安装 `wind-mcp-skill`，先向用户推荐安装，并说明缺少 Wind 数据源会影响分析结论的可靠性。

推荐安装命令：

```bash
npx skills add Wind-Information-Co-Ltd/wind-skills --skill wind-mcp-skill -g -y
```

若 GitHub 源不可用，使用 Gitee 镜像：

```bash
npx skills add https://gitee.com/wind_info/wind-skills.git --skill wind-mcp-skill -g -y
```

若已安装但不可用，应说明具体原因，例如缺少 `WIND_API_KEY`、CLI 不可执行、接口返回错误或额度不足，并先协助用户补齐数据条件。

若尚未取得足够的 Wind 数据，不应直接给出确定性结论；只能输出框架性判断、所需数据清单，或等待数据补齐后再分析。

## 技能定位

本技能用于解读公司管理层对未来经营指引的变动，重点回答“这次上修或下修究竟意味着什么、可信度高不高、影响会停留在情绪层还是会穿透到估值与逻辑层”。输出重点不是只翻译指引文字，而是把变动幅度、口径变化、背后动因和后续影响链条讲清楚。

## 执行流程

### Step 1：确认变动对象与原始口径

先确认本次变动的是哪类指引，以及基准口径是什么。优先厘清：

- 是收入、利润、利润率、现金流、资本开支还是订单指引
- 是全年指引、季度指引，还是阶段性经营展望
- 本次是正式调整，还是管理层在交流中边际修正表述
- 新旧口径是否可直接比较

如果口径有变化，必须先解释可比性，再讨论方向。

### Step 2：衡量变动幅度与市场预期差

把指引变化放到正确坐标里看，重点包括：

- 变动绝对幅度和相对幅度
- 与市场原有预期相比是超预期、符合还是不及
- 与公司历史沟通风格相比是激进、常规还是保守
- 当前估值是否已经部分计入这类调整

这一步的目标是判断“变化本身大不大”和“市场是否已经提前反应”。

### Step 3：拆解变动背后的驱动因素

进一步判断此次指引变化来自哪里，常见来源包括：

- 需求变化、价格变化、订单节奏变化
- 成本、费用、供应链或汇率扰动
- 新产品、新项目、新区域扩张的兑现或延迟
- 管理层风险偏好变化或沟通策略变化
- 一次性项目对短期口径造成影响

应优先抓主要驱动，不要把所有可能因素平均分配。

### Step 4：评估指引可信度与可持续性

同样的上修或下修，含金量可能完全不同。需要重点判断：

- 管理层过去给指引是否稳定可信
- 本次变化是否有足够经营证据支撑
- 是基于已确认订单，还是基于前瞻假设
- 哪些关键变量仍然脆弱，可能导致再次修正

### Step 5：映射到估值、预期与投资逻辑

把指引变化从“口径变化”翻译成“投资含义”，至少回答：

- 这会改变市场对增长、利润或质量的哪部分判断
- 估值锚点是否需要同步调整
- 原有看多或看空逻辑哪一边被强化
- 这次变化更像拐点确认、噪音扰动还是一次性修正

### Step 6：整合成变动解读成稿

最终输出应帮助使用者快速看懂“这次调指引有没有实质意义”，不展示中间搜集过程或内部草稿。

## 质量要求

1. 先解释变动口径和幅度，再下结论，避免跳步。
2. 必须区分“表述变化”与“经营趋势变化”。
3. 可信度评估不能省略，否则解读会失去判断价值。
4. 若新旧口径不可比，应明确指出，不做强行比较。
5. 结论要落到投资含义，而不是停留在管理层话术层面。
